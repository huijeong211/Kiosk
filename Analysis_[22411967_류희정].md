# 1. Introduction
## 1) Executive Summary
현재 카페 및 음료 매장에서 사용되는 대다수의 키오스크 시스템은 지나치게 복잡한 인터페이스와 불필요한 주문 단계로 인해 오히려 사용자에게 피로감을 주고 있다. 특히 기존 시스템들은 관리자(점주)의 편리함에만 치중하여, 정작 사용자가 직면하는 메뉴 탐색의 어려움이나 복잡한 옵션 선택 등의 문제를 방치하고 있다. 이로 인해 디지털 취약계층은 주문 과정에서 소외되고 있으며, 이는 매장 운영의 효율성 저하로 이어진다.

본 프로젝트에서 개발하는 Beverage Kiosk System은 이러한 기존 시스템의 한계를 극복하고자 한다. 복잡할수록 더 많은 기능을 제공할것이라는 잘못된 설계를 탈피하고, 인지 공학적 접근을 통해 사용자에게 가장 필요한 정보만을 직관적으로 노출함으로써 주문의 진입장벽을 획기적으로 낮춘 시스템을 구축하고자 한다.

## 2) Business Goals
Beverage Kiosk System 시스템의 목적은 사용자가 학습 없이도 즉시 주문을 완료할 수 있는 인터페이스를 제공하는 것이다.
기존 키오스크의 고질적인 문제인 복잡한 뎁스 대신 단 세 번의 터치만으로 결제까지 이어지는 프로세스를 구축하여 사용자 편의성과 매장 회전율을 극대화한다.
또 불필요한 회원가입이나 마케팅 팝업 등 사용자의 집중력을 흐트러뜨리는 요소를 과감히 제거하여 음료 주문이라는 본연의 목적에만 집중하게 함으로써 오주문율을 낮춘다. 이를 통해 매장은 인건비 절감을 넘어, 기술 취약 고객층까지도 아우르는 포용적 서비스를 실현한다.

## 3) Technical Goals
본 시스템은 안정적인 데이터 처리와 직관적인 사용자 경험을 위해 다음과 같은 기술적 목표를 가진다.

Security & Access Control: 재고 수량이나 메뉴 가격 같은 정보는 매우 중요하기 때문에 일반 사용자가 함부로 수정할 수 없도록 설계한다. 시스템 내에서 관리자 권한을 가진 사람만이 해당 데이터에 접근하고 수정할 수 있도록 관리 기능을 제한하여 정보의 안전성을 확보한다.

Algorithm & Performance: 사용자가 카테고리를 선택하거나 메뉴를 장바구니에 담을 때 지연 시간 없이 즉각적으로 실행되도록 설계한다. 각 기능의 특성에 맞는 적합한 알고리즘을 선택하여 데이터 처리 효율을 높이고, 사용자가 기다리는 불편함을 최소화한다.

User Interface (GUI): 기술적인 지식이 없는 사용자라도 화면만 보고 바로 주문을 진행할 수 있도록 누구나 이해하기 쉬운 직관적인 GUI를 제작한다. 불필요한 단계를 줄이고 버튼 위주의 레이아웃을 구성하여 사용자의 편의성을 최우선으로 한다.

## Use Case #3: Manage Cart

### GENERAL CHARACTERISTICS

| 항목 | 내용 |

| Scope | Kiosk System |
| Level | User Level |
| Author | Ryu HuiJeong |

| Status | Analysis |

### MAIN SUCCESS SCENARIO

| Step | Action |
|------|--------|

| 2 | 시스템이 해당 상품의 현재 수량 값을 확인한다. |

| 4 | 시스템이 변경된 수량을 기반으로 해당 상품의 금액을 재계산한다. |
| 5 | 시스템이 전체 장바구니 금액(총합)을 다시 계산한다. |

### GENERAL CHARACTERISTICS

## 3. Domain Analysis

### 1) Menu
메뉴의 이름, 가격, 설명, 이미지, 카테고리 등의 정보를 관리하는 클래스이다.  
사용자가 메뉴를 조회하거나 선택할 때 기본 데이터를 제공하며, 화면 출력에 필요한 핵심 정보의 단위가 된다.  
또한 Category와 연관되어 특정 분류 기준으로 메뉴를 필터링할 수 있도록 한다.

→ 관련 Use Case: Browse Menu, View Menu Detail

---

### 2) Category
메뉴를 카테고리(커피, 음료, 디저트 등) 기준으로 분류하여 관리하는 클래스이다.  
Menu 객체들을 그룹화하여 사용자에게 직관적인 탐색 환경을 제공하며, 선택된 카테고리에 따라 메뉴 목록을 필터링하는 역할을 수행한다.

→ 관련 Use Case: Browse Menu

---

### 3) Cart
사용자가 선택한 메뉴들을 임시로 저장하는 장바구니 클래스이다.  
CartItem 객체들을 포함하며, 전체 주문 금액 계산, 항목 추가/삭제, 수량 변경 등의 기능을 수행한다.  
결제 이전 단계에서 사용자 선택 상태를 유지하는 핵심 역할을 한다.

→ 관련 Use Case: Manage Cart, Cancel Order
Cart → CartItem 포함

---

### 4) CartItem
장바구니에 담긴 개별 메뉴 항목을 표현하는 클래스이다.  
각 항목은 메뉴 정보(Menu)와 수량(quantity), 부분 금액(subtotal)을 포함하며, Cart 내부에서 관리된다.  
수량 변경 시 해당 항목의 금액을 재계산하는 역할을 수행한다.

→ 관련 Use Case: Manage Cart

---

### 5) Order
사용자의 최종 주문 정보를 저장하는 클래스이다.  
결제 승인 이후 생성되며, 주문 번호, 주문 목록, 총 금액, 결제 상태 등의 정보를 포함한다.  
시스템에서 거래 단위로 처리되는 핵심 데이터 객체이다.

→ 관련 Use Case: Verify Payment
Order → CartItem 기반 생성

---

### 6) Payment
결제 요청 및 승인 처리 과정을 담당하는 클래스이다.  
주문(Order) 정보를 기반으로 결제 데이터를 생성하고, 외부 결제 시스템(PG)과 통신하여 승인 여부를 처리한다.  
결제 결과에 따라 주문 상태를 변경하는 역할을 수행한다.

→ 관련 Use Case: Process Payment, Verify Payment
Payment → Order 참조

---

### 7) Receipt
결제 완료 후 생성되는 영수증 데이터를 관리하는 클래스이다.  
주문 정보(Order)를 기반으로 상품 목록, 결제 금액, 승인 번호 등을 포함한 출력 데이터를 구성한다.  
프린터 출력 또는 화면 표시 기능과 연계된다.

→ 관련 Use Case: Print Receipt

---

### 8) User
시스템 사용자 정보를 관리하는 클래스이다.  
Customer(일반 사용자)와 Manager(관리자)로 구분되며, 각 사용자 유형에 따라 접근 가능한 기능이 달라진다.  
시스템 접근 및 권한 제어의 기준이 되는 객체이다.

→ 관련 Use Case: 전체 Use Case (Customer, Manager 역할 기반)

---

### 9) Inventory
상품의 재고 수량 및 품절 상태를 관리하는 클래스이다.  
각 Menu와 연관되어 현재 판매 가능 여부를 판단하며, 결제 완료 시 재고 차감, 관리자 수정 시 재고 갱신 기능을 수행한다.  
주문 처리와 관리자 기능 간 데이터 일관성을 유지하는 역할을 한다.

→ 관련 Use Case: Update Stock, Verify Payment

---

### 10) ProductManagement
관리자가 메뉴 정보를 생성, 수정, 삭제할 수 있도록 지원하는 클래스이다.  
Menu 객체에 대한 CRUD(Create, Read, Update, Delete) 작업을 수행하며, 변경된 데이터는 즉시 시스템에 반영된다.  
관리자 기능의 핵심 데이터 제어 역할을 담당한다.

→ 관련 Use Case: Manage Product

---

### 11) SalesReport
주문 데이터(Order)를 기반으로 매출 통계를 생성하는 클래스이다.  
총 매출액, 주문 건수, 상품별 판매량 등의 데이터를 집계하고, 이를 표 또는 그래프 형태로 제공한다.  
관리자가 운영 현황을 분석할 수 있도록 지원하는 역할을 수행한다.

→ 관련 Use Case: View Sales Report
