# 1. Introduction
## 1) Executive Summary
현재 카페 및 음료 매장에서 사용되는 대다수의 키오스크 시스템은 지나치게 복잡한 인터페이스와 불필요한 주문 단계로 인해 오히려 사용자에게 피로감을 주고 있다. 특히 기존 시스템들은 관리자(점주)의 편리함에만 치중하여, 정작 사용자가 직면하는 메뉴 탐색의 어려움이나 복잡한 옵션 선택 등의 문제를 방치하고 있다. 이로 인해 디지털 취약계층은 주문 과정에서 소외되고 있으며, 이는 매장 운영의 효율성 저하로 이어진다.

본 프로젝트에서 개발하는 Beverage Kiosk System은 이러한 기존 시스템의 한계를 극복하고자 한다. 복잡할수록 더 많은 기능을 제공할것이라는 잘못된 설계를 탈피하고, 인지 공학적 접근을 통해 사용자에게 가장 필요한 정보만을 직관적으로 노출함으로써 주문의 진입장벽을 획기적으로 낮춘 시스템을 구축하고자 한다.

## 2) prominent features
### (1) Usefulness
본 시스템은 사용자의 실제 주문 과정에서 발생하는 불편을 최소화하는 데 초점을 둔다.
카테고리 선택, 메뉴 탐색, 장바구니 추가 및 결제 과정까지 모든 기능은 지연 없이 즉각적으로 반응하도록 설계한다. 이를 위해 기능별로 적절한 알고리즘을 적용하여 데이터 처리 속도를 최적화하고, 사용자가 별도의 학습 없이도 직관적으로 사용할 수 있는 GUI를 제공한다.
또한 불필요한 입력 절차를 제거하고 버튼 중심의 인터페이스를 구성하여 주문 과정을 단순화한다.

### (2) Significance
키오스크 환경에서는 잘못된 정보나 시스템 오류가 곧바로 매출 손실과 사용자 불만으로 이어진다.
따라서 메뉴 가격, 재고 데이터 등 핵심 정보는 관리자만 접근 및 수정할 수 있도록 권한을 엄격히 제한하여 시스템의 신뢰성과 데이터 무결성을 확보한다.
또한 직관적인 인터페이스 설계를 통해 디지털 기기에 익숙하지 않은 사용자도 쉽게 이용할 수 있도록 하여, 기술 소외 계층까지 포함하는 포용적 서비스 환경을 구축한다.

### (3) Expandability
본 시스템은 향후 기능 확장을 고려하여 설계된다.
메뉴 데이터, 사용자 인터페이스, 결제 기능 등을 모듈화하여 새로운 메뉴 추가, 결제 방식 확장(예: 간편결제), 또는 다국어 지원 기능 등을 쉽게 적용할 수 있도록 한다.
또한 관리자 기능 역시 확장 가능하게 설계하여 향후 매출 분석, 재고 자동 관리 등 추가적인 운영 기능으로 발전할 수 있는 기반을 마련한다.

| 7 | 시스템이 필터링된 메뉴 리스트를 이미지, 이름, 가격과 함께 화면에 표시한다. |

## Use Case #3: Manage Cart

### GENERAL CHARACTERISTICS

| 항목 | 내용 |
|------|------|
| Summary | 사용자가 장바구니에 담긴 메뉴의 수량을 조절하거나 항목을 삭제하며, 변경된 수량에 따라 총 결제 금액을 실시간으로 확인한다. 시스템은 사용자 입력에 따라 데이터 상태를 즉시 반영하고 화면에 결과를 업데이트한다. |
| Scope | Kiosk System |
| Level | User Level |
| Author | Ryu HuiJeong |
| Last Update | 2026-04-25 |
| Status | Analysis |
| Primary Actor | Customer |
| Preconditions | 장바구니에 최소 하나 이상의 상품이 추가된 상태여야 하며, 각 상품은 유효한 가격 및 재고 정보를 가지고 있어야 한다. |
| Trigger | 사용자가 장바구니 화면에서 수량 조절 버튼(+, -) 또는 삭제 기능을 선택한다. |
| Success Post Condition | 변경된 수량이 즉시 반영되며, 개별 상품 금액 및 총 결제 금액이 재계산되어 화면에 표시된다. |
| Failed Post Condition | 수량 변경이 불가능한 경우(재고 초과 등) 사용자에게 안내 메시지가 표시되고 기존 상태가 유지된다. |

---

### MAIN SUCCESS SCENARIO

| Step | Action |
|------|--------|
| 1 | 사용자가 장바구니 화면에서 특정 상품 옆의 '+' 또는 '-' 버튼을 선택한다. |
| 2 | 시스템이 해당 상품의 현재 수량 값을 확인한다. |
| 3 | 시스템이 사용자 입력에 따라 수량을 증가 또는 감소시킨다. |
| 4 | 시스템이 변경된 수량을 기반으로 해당 상품의 금액을 재계산한다. |
| 5 | 시스템이 전체 장바구니 금액(총합)을 다시 계산한다. |
| 6 | 변경된 수량과 금액을 화면에 즉시 반영한다. |

## Use Case #4: Process Payment

### GENERAL CHARACTERISTICS

# 3. Domain Analysis

## 1) Menu
메뉴의 이름, 가격, 설명, 이미지, 카테고리 등의 정보를 관리하는 클래스이다.  
사용자가 메뉴를 조회하거나 선택할 때 기본 데이터를 제공하며, 화면 출력에 필요한 핵심 정보의 단위가 된다.  
또한 category와 연관되어 특정 분류 기준으로 메뉴를 필터링할 수 있도록 한다.

→ 관련 Use Case: Browse Menu, View Menu Detail

---

## 2) Category
메뉴를 카테고리(커피, 음료, 디저트 등) 기준으로 분류하여 관리하는 클래스이다.  
Menu 객체들을 그룹화하여 사용자에게 직관적인 탐색 환경을 제공하며, 선택된 카테고리에 따라 메뉴 목록을 필터링하는 역할을 수행한다.

→ 관련 Use Case: Browse Menu

---

## 3) Cart
사용자가 선택한 메뉴들을 임시로 저장하는 장바구니 클래스이다.  
CartItem 객체들을 포함하며, 전체 주문 금액 계산, 항목 추가/삭제, 수량 변경 등의 기능을 수행한다.  
결제 이전 단계에서 사용자 선택 상태를 유지하는 핵심 역할을 한다.

→ 관련 Use Case: Manage Cart, Cancel Order
Cart → CartItem 포함

---

## 4) CartItem
장바구니에 담긴 개별 메뉴 항목을 표현하는 클래스이다.  
각 항목은 메뉴 정보와 수량, 부분 금액을 포함하며, Cart 내부에서 관리된다.  
수량 변경 시 해당 항목의 금액을 재계산하는 역할을 수행한다.

→ 관련 Use Case: Manage Cart

---

## 5) Order
사용자의 최종 주문 정보를 저장하는 클래스이다.  
결제 승인 이후 생성되며, 주문 번호, 주문 목록, 총 금액, 결제 상태 등의 정보를 포함한다.  
시스템에서 거래 단위로 처리되는 핵심 데이터 객체이다.

→ 관련 Use Case: Verify Payment
Order → CartItem 기반 생성

---

## 6) Payment
결제 요청 및 승인 처리 과정을 담당하는 클래스이다.  
주문 정보를 기반으로 결제 데이터를 생성하고, 외부 결제 시스템(PG)과 통신하여 승인 여부를 처리한다.  
결제 결과에 따라 주문 상태를 변경하는 역할을 수행한다.

→ 관련 Use Case: Process Payment, Verify Payment
Payment → Order 참조

---

## 7) Receipt
결제 완료 후 생성되는 영수증 데이터를 관리하는 클래스이다.  
주문 정보를 기반으로 상품 목록, 결제 금액, 승인 번호 등을 포함한 출력 데이터를 구성한다.  
프린터 출력 또는 화면 표시 기능과 연계된다.

→ 관련 Use Case: Print Receipt

---

## 8) User
시스템 사용자 정보를 관리하는 클래스이다.  
Customer(고객)와 Manager(관리자)로 구분되며, 각 사용자 유형에 따라 접근 가능한 기능이 달라진다.  
시스템 접근 및 권한 제어의 기준이 되는 객체이다.

→ 관련 Use Case: 전체 Use Case (Customer, Manager 역할 기반)

---

## 9) Inventory
상품의 재고 수량 및 품절 상태를 관리하는 클래스이다.  
각 Menu와 연관되어 현재 판매 가능 여부를 판단하며, 결제 완료 시 재고 차감, 관리자 수정 시 재고 갱신 기능을 수행한다.  
주문 처리와 관리자 기능 간 데이터 일관성을 유지하는 역할을 한다.

→ 관련 Use Case: Update Stock, Verify Payment

---

## 10) ProductManagement
관리자가 메뉴 정보를 생성, 수정, 삭제할 수 있도록 지원하는 클래스이다.  
Menu 객체에 대한 생성, 업데이트, 삭제 등의 작업을 수행하며, 변경된 데이터는 즉시 시스템에 반영된다.  
관리자 기능의 핵심 데이터 제어 역할을 담당한다.

→ 관련 Use Case: Manage Product

---

## 11) SalesReport
주문 데이터를 기반으로 매출 통계를 생성하는 클래스이다.  
총 매출액, 주문 건수, 상품별 판매량 등의 데이터를 집계하고, 이를 표 또는 그래프 형태로 제공한다.  
관리자가 운영 현황을 분석할 수 있도록 지원하는 역할을 수행한다.

→ 관련 Use Case: View Sales Report

각 항목은 CartItem 객체로 관리되며, 사용자는 수량을 조절하거나 항목을 삭제할 수 있다.
시스템은 수량 변경에 따라 총 결제 금액을 실시간으로 재계산하여 표시한다.
사용자는 ‘결제하기’를 통해 결제 단계로 이동할 수 있다.

## 5) Process Payment
<img width="433" height="601" alt="5번" src="https://github.com/user-attachments/assets/5baea783-bf5b-4d90-a874-6208bb6db4c9" />

결제 화면에서는 사용자가 선택한 상품들의 총 금액이 표시되며, 다양한 결제 수단을 선택할 수 있다.
사용자가 결제를 진행하면 시스템은 Payment 객체를 통해 결제 요청을 처리하고, 외부 결제 시스템과 연동하여 승인 절차를 수행한다.
결제가 완료되면 주문이 생성되고 다음 단계로 이동한다.

## 6) Print Receipt
<img width="433" height="598" alt="6번" src="https://github.com/user-attachments/assets/d4ea47d7-2c12-4c69-a7c4-2a6ac096e386" />

결제 완료 후 영수증 화면이 출력된다.
해당 화면에서는 주문 번호, 주문 내역, 총 결제 금액 등의 정보가 표시되며, 이는 Receipt 객체를 기반으로 생성된다.
사용자는 주문 결과를 확인할 수 있으며, 시스템은 전체 주문 프로세스를 종료한다.

## 7) Cancel Order
<img width="416" height="369" alt="7번" src="https://github.com/user-attachments/assets/b034a339-9092-494e-8b94-f5d67f4d96fe" />

주문 취소 화면에서는 사용자가 현재 진행 중인 주문을 취소할 수 있다.
취소 요청 시 확인 메시지가 출력되며, 사용자가 ‘예’를 선택하면 장바구니(Cart)와 관련된 모든 데이터가 초기화된다.
이후 시스템은 메인 화면으로 돌아가며, ‘아니오’를 선택할 경우 기존 상태가 유지된다.

## 8) Product Management (Admin Mode)
<img width="470" height="593" alt="8번" src="https://github.com/user-attachments/assets/8187d679-5214-4853-bd17-7cc3703c4c7c" />

관리자 모드에서는 상품 관리 기능이 제공된다.
관리자는 상품(Menu)의 추가, 수정, 삭제 작업을 수행할 수 있으며, 이러한 작업은 ProductManagement 객체를 통해 처리된다.
입력된 데이터는 검증 과정을 거쳐 시스템에 반영되며, 변경된 내용은 사용자 화면에도 즉시 적용된다.

# 5. Glossary

| Term | Definition |
|------|------------|
| Kiosk System | 사용자가 직원 없이 직접 주문 및 결제를 수행할 수 있도록 제공되는 무인 주문 시스템 |
| Customer | 키오스크를 이용하여 메뉴를 탐색하고 주문 및 결제를 수행하는 일반 사용자 |
| Manager | 상품 관리, 재고 관리, 매출 조회 등의 관리자 기능을 수행하는 사용자 |
| Menu | 음료 또는 상품의 기본 정보(이름, 가격, 설명, 이미지 등)를 포함하는 데이터 단위 |
| Category | 메뉴를 유형별로 분류한 그룹 (예: 커피, 에이드, 디저트 등) |
| Cart | 사용자가 선택한 상품들을 임시로 저장하는 장바구니 |
| CartItem | 장바구니에 포함된 개별 상품 항목으로, 메뉴 정보와 수량 및 금액을 포함 |
| Order | 결제가 완료된 후 생성되는 최종 주문 데이터 (주문 번호, 상품 목록, 총 금액 등 포함) |
| Payment | 결제 처리 과정 및 외부 결제 시스템과의 연동을 담당하는 기능 |
| Payment Gateway (PG) | 카드 및 모바일 결제를 처리하는 외부 결제 시스템 |
| Verify Payment | 결제 승인 여부를 확인하고 주문을 확정하는 과정 |
| Receipt | 결제 완료 후 생성되는 영수증 데이터 (상품 정보, 금액, 승인 번호 등 포함) |
| Inventory | 각 상품의 재고 수량 및 품절 상태를 관리하는 데이터 |
| Product Management | 관리자가 메뉴 정보를 추가, 수정, 삭제하는 기능 |
| Sales Report | 주문 데이터를 기반으로 생성된 매출 통계 정보 |
| Session | 사용자 주문 진행 상태(장바구니 등)를 일시적으로 저장하는 시스템 상태 |
| Transaction | 결제 및 주문 처리 과정에서 데이터 일관성을 보장하기 위한 처리 단위 |
| Use Case | 시스템이 사용자와 상호작용하여 특정 기능을 수행하는 시나리오 |
| Include | 하나의 Use Case가 다른 Use Case를 항상 포함하여 수행하는 관계 |
| Extend | 특정 조건에서만 추가적으로 수행되는 확장 시나리오 관계 |
| UI (User Interface) | 사용자가 시스템과 상호작용하는 화면 구성 요소 |
| Main Screen | 시스템 시작 시 사용자에게 처음 제공되는 초기 화면 |
| Admin Mode | 관리자만 접근 가능한 기능 영역 |
| Error Message | 시스템 오류 발생 시 사용자에게 안내되는 메시지 |
| Data Validation | 입력 데이터의 유효성을 검사하는 과정 |
| Concurrency | 여러 사용자 또는 작업이 동시에 실행될 때 발생하는 처리 상황 |
